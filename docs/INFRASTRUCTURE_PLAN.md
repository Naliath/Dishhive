# Dishhive — Infrastructure & Setup Plan

**Created:** April 19, 2026  
**Status:** 🚧 In Progress

---

## 1. Overview

Dishhive is a household meal planning application built on the same platform approach as **Freezy** (FreezerInventory). It shares ecosystem conventions but runs as an independent application with its own data store, ports, and configuration.

### Goals of This Document
- Define the full infrastructure before any functional feature implementation
- Provide a reference for local development and production deployment
- Document integration points with Freezy
- Record all assumptions and open questions

## 4. Local Port Allocation

Dishhive uses different ports than Freezy to allow both apps to run simultaneously.

| Service         | Freezy (FreezerInventory) | Dishhive |
|-----------------|--------------------------|----------|
| App (HTTP)      | 5000                     | **5100** |
| App (HTTPS dev) | 5001                     | **5101** |
| PostgreSQL      | 5432                     | **5433** |
| Angular dev server | 4200                 | **4201** |

## 5. Backend Setup

### 5.1 .NET Project

- **Solution:** `Dishhive.sln`
- **API project:** `Dishhive.Api` (SDK: `Microsoft.NET.Sdk.Web`)
- **Tests project:** `Dishhive.Api.Tests` (SDK: `Microsoft.NET.Sdk`)
- **Target framework:** `net10.0`
- **Nullable enabled**, **ImplicitUsings enabled**

### 5.2 Database Context

- `DishhiveDbContext` inheriting `DbContext`
- All primary keys are `Guid` with default `gen_random_uuid()`
- `CreatedAt` and `UpdatedAt` use `CURRENT_TIMESTAMP` defaults
- `UpdatedAt` is automatically set via `SaveChanges` override

### 5.3 Migrations

Run from `src/Dishhive.Api/`:
```bash
dotnet ef migrations add InitialCreate
dotnet ef database update
```

### 5.4 Configuration

Connection string in `appsettings.json`:
```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Host=localhost;Port=5433;Database=dishhive;Username=dishhive;Password=dishhive_dev_password"
  }
}
```

Override per environment via environment variables:
```
ConnectionStrings__DefaultConnection=...
```

### 5.5 API Structure

All endpoints are prefixed with `/api/`. Controllers follow RESTful conventions. See `docs/dotnet/api-standards.md` for details.

### 5.6 Health Check

- Endpoint: `GET /health`
- Checks database connectivity in non-test environments

## 6. Frontend Setup

### 6.1 Angular Project

- **Project name:** `dishhive-web`
- **Output path:** `dist/dishhive-web/browser` → copied to `wwwroot` in Docker
- **Dev server port:** 4201
- **Proxy:** `/api/*` → `http://localhost:5100`

### 6.2 Version Management

Follows same `package.json`-as-source-of-truth pattern as Freezy:
- `scripts/sync-version.js` propagates version to `AppVersion.cs` and `manifest.webmanifest`
- Run via `npm run build` (pre-build step)

### 6.3 App Structure

Angular standalone components with lazy-loaded routes. No NgModules. Change detection: `OnPush` preferred for performance-sensitive components.

Initial routes:
- `/` → Week planner (home/default view)
- `/family` → Family composition
- `/recipes` → Recipe store
- `/settings` → Settings (including measurement preferences)

## 7. Docker Setup

### 7.1 docker-compose.yml

Two services:
- `dishhive-db`: PostgreSQL 16, port 5433, volume `dishhive_postgres_data`
- `dishhive-app`: App container, port 5100, depends on `dishhive-db`

### 7.2 Dockerfile

Multi-stage build identical to Freezy's pattern:
1. **Stage 1 (frontend-build):** Node 22 Alpine — `npm ci && npm run build`
2. **Stage 2 (backend-build):** .NET SDK 10 — `dotnet publish`
3. **Stage 3 (runtime):** .NET ASP.NET 10 runtime, non-root user, health check via curl

### 7.3 Commands

```bash
# Start all services
docker-compose up -d

# Rebuild after code changes
docker-compose up -d --build

# View logs
docker-compose logs -f

# Stop and remove volumes
docker-compose down -v
```

## 9. API Contracts Strategy

- Controllers return DTOs only — entities are never serialized directly
- Request validation via `[Required]`, `[MaxLength]`, etc. on DTO properties
- Standard response codes: 200, 201, 204, 400, 404, 500
- All responses use camelCase JSON (ASP.NET Core default)
- Swagger/OpenAPI always enabled (useful for self-hosted apps)

## 10. Freezy Integration Architecture

Dishhive integrates with Freezy to allow frozen leftovers/meals to be included in meal planning.

### Integration Approach (Phase 1): API-based read integration

- Dishhive backend calls Freezy's API to query frozen items
- Integration is one-directional: Dishhive reads from Freezy
- The Freezy API endpoint(s) to use are documented in `docs/features/04-freezy-integration.md`
- Connection details (base URL) are configurable via `appsettings.json`

### Configuration

```json
{
  "FreezyIntegration": {
    "BaseUrl": "http://localhost:5000",
    "Enabled": true
  }
}
```

### Isolation Guarantee

The Dishhive codebase contains no direct database access to Freezy's PostgreSQL instance. All communication goes through the Freezy HTTP API. This ensures:
- Schema changes in Freezy do not break Dishhive directly
- The integration can be disabled without breaking Dishhive's core features

## 12. Assumptions

1. **Angular version** matches Freezy (Angular 21) to stay consistent and allow potential shared tooling in the future.
2. **PostgreSQL version** matches Freezy (v16) for operational consistency.
3. **No shared database** between Freezy and Dishhive — they each have their own instance.
4. **Freezy runs locally** when Freezy integration features are tested or used.
5. **No authentication system** is in scope for initial Dishhive implementation. This aligns with Freezy's current state (a user auth system is only planned, not implemented, in Freezy).
6. **Metric system** is the default for measurement preferences.
7. **Dutch-language import** is supported as a primary use case given the VRT Dagelijkse Kost source.

## 13. Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| VRT Dagelijkse Kost scraping breaks due to HTML changes | Medium | Medium | Pluggable source-provider architecture; tests verify extraction |
| Freezy API changes break Dishhive integration | Low | Medium | Integration isolated behind a service interface; configurable base URL |
| Port conflicts on developer machines | Low | Low | Clearly documented; all ports configurable |
| EF Core migration drift between developers | Low | Medium | Migrations committed to source control |
| .NET 10 or Angular 21 minor breaking changes | Low | Low | Pin major versions; update intentionally |
